# Design: Build It With a Point of View

If it could have been generated by a default prompt, it is not good enough.

## Lock the Direction First

Before writing any code, answer these from the product context. If a point is genuinely ambiguous, ask the user one concise question:

1. **Who uses this, and in what context?**
2. **What is the aesthetic direction?** Name it precisely.
3. **What is the one thing this leaves in memory?**
4. **What are the hard constraints?** Framework, bundle size, accessibility, contrast, motion limits.

Do not proceed until all four are answered. State the chosen direction in one sentence. Then load `references/design-reference.md` and check the tech stack conflicts table. Name the single CSS strategy before writing the first component.

Before writing any code, write three things:
- **Visual thesis**: one sentence describing mood, material, and energy
- **Content plan**: hero → support → detail → final CTA, one line each
- **Interaction thesis**: 2-3 specific motion ideas that change how the page feels

For app/dashboard surfaces: skip the marketing structure. Default to utility mode: orient the user, show status, enable action.

## Non-Negotiable Constraints

- **Typography**: no Inter, Roboto, or system-ui as the primary display typeface.
- **Color**: one color owns the page, one or two accents cut against it.
- **Motion**: ship a small set of intentional motions. If removing them changes nothing about the feel, remove them.
- **Layout**: default to cardless sections.
- **Backgrounds**: use gradients, texture, or layered depth. Flat defaults are rarely right.
- **Spatial composition**: asymmetry, overlap, diagonal flow, or grid-breaking elements.
- **Accessibility baseline**: honor `prefers-reduced-motion`, visible focus states, `alt` text on images.

## Gotchas

| What happened | Rule |
|---------------|------|
| Used Inter as the display font | It communicates nothing. Pick something with personality. |
| Three cards, identical shadows, identical padding | If swapping content does not require layout changes, redo it. |
| Claimed it looked right without opening a browser | Code correct in your head can look broken in the browser. Open it. |
| Chose glassmorphism, ignored the mobile constraint | Name the tradeoff. |
| Colors that "go with everything" go with nothing | Commit to a dominant color plus one sharp accent. |
| Dashboard built like a marketing page | App surfaces need utility copy and a working surface first. |
| Every project ended up with the same look | Vary light/dark, serif/sans, dense/spacious. |

## Handoff

Run these litmus checks before the summary:
- Is the first screen unmistakable?
- Is there one strong visual anchor?
- Can the page be understood by scanning headlines only?
- Does each section have one job?
- Does motion improve hierarchy or atmosphere?

If any check fails, fix it first. Then reload `references/design-reference.md` and check the common traps list. Then ask the user to inspect the result at full width and at 375px width. If the layout breaks, fix it before handing off.

End with:
- Aesthetic direction, named and justified in 2-3 sentences
- Non-obvious choices explained: typeface, color, layout logic
- Instructions for replacing placeholder content with real content
