# 编码注意事项

- 注释编写原则：
  - 注释优先使用中文，必要的技术术语可保留英文；
  - 注释是代码表达能力不足时的补充，不是替代品；
  - 优先通过清晰命名、合理拆分和良好结构表达代码意图；
  - 注释重点说明设计原因、业务约束、边界条件和风险点，着重写代码本身看不出来的信息。

- 避免过度设计：
  - 以满足当前业务需求为主，拒绝为了“未来可能用到”而提前堆砌复杂抽象；
  - 在保证可扩展性的前提下，优先选择简洁、直接、易维护的实现方案。

- 对不熟悉的技术栈或 API：
  - 不凭经验猜测，不照印象写；
  - 先查官方文档、权威资料或项目既有实现，再进行编码；
  - 避免因误用机制、错误假设或版本差异导致返工。

- 代码质量要求：
  - 代码首先是写给人看的，其次才是给机器执行的；
  - 保持代码清晰、可读、可维护，避免职责混乱、命名含糊、逻辑堆叠；
  - 编写代码时遵循《代码整洁之道》的核心思想，但不机械套用规则，以实际可读性和可维护性为准。

- 不要造轮子，优先考虑有没有实现好的库，可以借助网络搜索工具检索

# 特别注意
重要：直奔主题。先用最简单的方法，不要兜圈子。不要过度展开。务必极其简洁。

保持输出简短、直接。先给答案或行动，不要先讲理由。省去套话、铺垫和不必要的过渡。不要重复用户的话——直接执行。解释时，只保留用户理解所必需的内容。

文本输出应重点放在：
  - 需要用户决定的事项
  - 关键节点上的高层级进展更新
  - 会改变计划的错误或阻碍

如果一句话能说清，就不要说三句。尽量用短句、直接表达，不要长篇解释。代码或工具调用不受此限制。
